## 1. Project Overview

GridSync is an AI-powered Alberta power grid coordination platform. It:
1. Fetches **real-time grid demand** from the AESO API every 60 seconds.
2. Predicts grid stress **24–48 hours ahead** using a trained Facebook Prophet ML model.
3. Allows operators to send **real-time emergency alerts** to all consumer dashboards via Supabase Realtime WebSockets.
4. Shows consumers actionable tips and a live demand gauge so they can reduce usage during critical periods.

**Live URL**: [https://gridsync-ten.vercel.app/](https://gridsync-ten.vercel.app/)  
**Repo**: [github.com/Manrajwazir/gridsync-](https://github.com/Manrajwazir/gridsync-)

## 5. Critical Constants

The Alberta grid's safe capacity ceiling is **13,000 MW** (based on Dec 2025 record peak of 12,785 MW).

| Constant | Value | Used In |
|:---|:---|:---|
| `MAX_CAPACITY` | `13000` | `live-usage.js`, `historical-usage.js`, `GridStatus.js` |
| Critical threshold | `>12200 MW` | `live-usage.js`, `predict.py` |
| Warning threshold | `>11200 MW` | `live-usage.js`, `predict.py` |
| DemandGauge default `maxMw` | `13000` | `DemandGauge.js` |
| Chart Y-axis domain | `[8000, 14000]` | `GridStatus.js` |
| Accuracy formula | `100 - (MAE / 13000) * 100` | `operator/index.js`, `operator/analytics.js` |

```

                      ▼

```

## 7. Authentication & Routing

### Auth System
- **Provider**: Supabase Auth (email/password)
- **Hook**: `hooks/useUser.js` returns `{ user, role, loading, signOut }`
- **Role resolution**: Checks `profiles` table first, falls back to `user_metadata.role`, defaults to `'consumer'`

### Supabase Tables
1. **`alerts`** — Real-time alert system
   - Columns: `id (uuid)`, `created_at`, `message (text)`, `type (text)`, `is_active (boolean)`
   - RLS: SELECT open to all, INSERT/UPDATE/DELETE require authenticated role
   - Realtime enabled via `alter publication supabase_realtime add table alerts`

2. **`profiles`** — User roles
   - Columns: `id (uuid, FK to auth.users)`, `role (text: 'operator' | 'consumer')`
   - Used by `useUser.js` to determine layout and access

### Routing Logic (`_app.js`)
- `/login` → No navigation, bare login page
- `/operator/*` → Requires auth + `role === 'operator'`, uses sidebar layout (`OperatorNav`)
- Everything else → Consumer layout with bottom tabs (`ConsumerNav`), no auth required to view

## 8. Key Component Details

### `GridStatus.js` (THE main component)
- Fetches `/api/live-usage` every 60 seconds
- Fetches `/api/historical-usage` for last 24h actuals
- Loads `/predictions.json` for 48h forecast
- Merges data using a Map keyed by label to avoid duplicates
- Chart: `ComposedChart` with solid blue "Actual" line, dashed cyan "Predicted" line, shaded Area for confidence band
- Shows capacity bar, status badge (STABLE/ELEVATED/CRITICAL), countdown to next risk period
- Calls `onReady(stats)` to pass live data up to parent components (pct, status, usageMw)

### `DemandGauge.js`
- Pure SVG arc gauge (no external libs)
- Props: `pct`, `status`, `usageMw`, `maxMw` (default 13000)
- Color-coded: green (STABLE) → amber (WARNING) → red (CRITICAL)

### `AlbertaMap.js`
- Dynamically loads Leaflet via CDN to avoid SSR issues
- CartoDB Dark Matter tiles (free, no API key)
- 10 Alberta cities with color-coded risk markers
- Zoom controls positioned at `bottomright`
- Click a city for a popup with risk status

### Consumer Home (`pages/index.js`)
- Two-column layout: GridStatus chart on left, DemandGauge + collapsible tips on right
- **Real-time alert listener**: subscribes to `postgres_changes` on `alerts` table
  - Shows red pulsing siren banner when `is_active === true`
  - Clears banner instantly when alert is deactivated (supports both INSERT and UPDATE events)

### Operator Alerts (`pages/operator/alerts.js`)
- **Send Alert**: Inserts into `alerts` table with `is_active: true`
  - **Duplicate prevention**: Blocked if any alert already has `is_active: true`
- **Cancel Alert**: Updates all active alerts to `is_active: false`
  - **Empty prevention**: Blocked if no active alerts exist
- Real-time listener keeps the alert history table in sync
- Alert history: real data from Supabase, mock engagement metrics for scale

### Operator Dashboard (`pages/operator/index.js`)
- Quick stats: Active Users (jittered), MW Saved (0.15% of live load), Alerts Sent (real count from last 24h), Model Accuracy (from accuracy.json)
- Two-column: GridStatus chart left, AlbertaMap + DemandGauge right

## 9. ML Pipeline

### Training Flow
```
Raw AESO CSV → clean_data.py → grid_clean.csv
Open-Meteo API → download_weather.py → weather_clean.csv
grid_clean.csv + weather_clean.csv → merge_and_train.py → alberta_model.pkl + accuracy.json
```

### Prediction Flow (`model/predict.py`)
1. Loads `model/alberta_model.pkl`
2. Fetches 3-day weather forecast from Open-Meteo (Edmonton coords: 53.5461, -113.4938)
3. Builds 48-hour future DataFrame starting from current hour
4. Adds `temperature_c` (with -10°C fallback) and `is_weekend` features
5. Runs `model.predict(future)`
6. Maps predictions to risk levels: `>12200 MW = critical`, `>11200 MW = warning`, else `safe`
7. Writes output to `public/predictions.json`

### Prophet Model Configuration
- **Algorithm**: Facebook Prophet with piecewise linear trend
- **Seasonality mode**: Multiplicative
- **Seasonality layers**: Daily (P=24h), Weekly (P=168h), Yearly (P=8760h)
- **Regressors**: `temperature_c` (standardized), `is_weekend` (binary)
- **Changepoint prior scale**: 0.05
- **Training data**: ~8,423 rows of hourly AESO grid data
- **Accuracy**: MAE ≈ 184 MW, 94.2% within ±200 MW, 99.1% within ±500 MW

### GitHub Action (`daily-repredict.yml`)
- Cron: `0 6 * * *` (06:00 UTC daily)
- Also supports manual dispatch via `workflow_dispatch`
- Steps: checkout → setup Python 3.10 → install requirements.txt → run `model/predict.py` → commit & push updated `predictions.json`
- Commit message: `🤖 Auto-update predictions [skip ci]`

## 10. API Endpoints

### `GET /api/live-usage`
- **Source**: AESO Current Supply Demand API v2
- **Headers**: `API-Key: {AESO_API_KEY}`
- **Response**: `{ usage_mw, capacity_percent, last_updated, status, is_mock? }`
- **Fallback**: Returns mock data (10500 MW) if no API key or request fails
- **Status logic**: `>12200 = CRITICAL`, `>11200 = WARNING`, else `STABLE`

### `GET /api/historical-usage?anchor=<MW>`
- **Source**: AESO Pool Price Report API (tries v1.1, v1, and alternative paths)
- **Response**: Array of `{ timestamp, usage_mw }` for last 24 hours
- **Fallback**: Generates plausible mock curve anchored to the `anchor` param (default 11200)

## 12. Deployment Notes

- **Vercel**: Auto-deploys on push to `main`. Set all 3 env vars in Vercel project settings.
- **Supabase**: Run `supabase_setup.sql` in the SQL Editor to create the `alerts` table with RLS + Realtime.
- **Profiles table**: Must exist in Supabase with at least one row per user (`id`, `role`). Create operator accounts by setting `role = 'operator'` in the profiles table.
